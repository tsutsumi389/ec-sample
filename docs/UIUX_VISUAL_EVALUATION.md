# UI/UX 評価基準（実画面スクリーンショットベース・100点満点）

対象: 本アプリの実際の画面（ブラウザでレンダリングされた状態）。ゲスト / 一般ユーザー / 管理者の3状態、デスクトップ（1280px）とモバイル（375px）の2ビューポートで撮影したスクリーンショットを主対象とする。最低限、以下のページ・状態をカバーすること。

- ゲスト: トップ、商品一覧、商品詳細、カート（空 or 未ログイン時の挙動）、ログイン、会員登録、404、モバイルでのヘッダー表示
- 一般ユーザー: ログイン後トップ、カート（商品あり／バリデーションエラー時）、商品カートへ追加直後、注文履歴一覧、注文詳細、注文完了
- 管理者: ダッシュボード／商品管理／注文管理／ユーザー管理の一覧、商品新規作成・編集モーダル

合格ライン: **80点以上**。

評価方法: 上記スクリーンショットの目視確認を主とする「実画面ベース」の評価。`docs/UIUX_EVALUATION.md`（コードレビューによる静的評価）とは別軸であり、両者は独立に採点する。ただし以下の項目は静的画像だけでは判定できないため、該当箇所のソースコードを確認して採点する（表中に「コードで確認」と明記）。

- ローディング状態（スピナー・スケルトン表示の有無）
- 多重送信防止（送信中のボタン disabled 化）
- フォーカスリング（`:focus-visible` 等の可視化）
- セマンティックHTML・`alt`・ARIA属性
- 破壊的操作の確認ダイアログの実装有無

## 採点ルール

- 各項目は 0〜配点の整数で採点する。部分的に満たす場合は按分してよい。
- **根拠必須**: 採点にはスクリーンショットのファイル名（例: `guest-products--mobile.png`）を必ず引用すること。コードで確認する項目は、ファイルパス・行番号を引用すること。根拠を示せない加点は禁止。
- **「一部のページ・状態のみ対応」は配点の半分を上限とする**。例えば「デスクトップでは崩れないがモバイルでは崩れる」「ゲストのみ確認できたが会員/管理者では未確認」といったケースは、その項目の配点の50%を超えて加点してはならない。
- 撮影対象に含まれないページ・状態がある場合は、確認できた範囲内で採点し、評価コメントに「未確認: ◯◯」を明記する。
- 減点方式ではなく「満たしている度合い」で加点する（0点＝全く満たさない、満点＝完全に満たす）。

## 1. 第一印象・ビジュアルデザイン（20点）

| 項目 | 配点 | 判定基準 |
|---|---|---|
| レイアウトの美しさ | 4 | 主要ページ（トップ・一覧・詳細）が余白のある整ったグリッドで構成され、雑然とした印象がない。要素が中央寄せ/左寄せで意図的に揃っている |
| 視覚的階層 | 3 | ページタイトル（`商品一覧`等の見出し）→価格・在庫等の重要情報→補足情報の順にフォントサイズ・太さ・色でメリハリがついている。価格が最も目立つ色・太さで表示されているか |
| 色彩・タイポグラフィ | 4 | プライマリカラー（ボタン・リンク色）がページ間で統一されている。見出し・本文のフォントサイズ差が一貫している。装飾色が乱用されていない |
| 商品画像の見え方（壊れた画像・プレースホルダの品質） | 5 | 商品画像が正しく表示されるか、少なくとも「画像なし」であることが分かる意図的なプレースホルダ（アイコン・グレー背景等）になっているかを確認する。ブラウザの壊れた画像アイコンや代替テキストの文字化け（`?` の連続表示等）がそのまま見えている場合は大幅減点 |
| 余白・整列 | 2 | カード内・セクション間の余白が均一。要素の左右端が揃っている（微妙なズレがない） |
| ブランド感 | 2 | ロゴ・サイト名・配色に統一感があり「作りかけ」感がない。汎用テンプレートのままの印象が強い場合は減点 |

## 2. ナビゲーション・情報設計（15点）

| 項目 | 配点 | 判定基準 |
|---|---|---|
| 現在地の把握（アクティブ表示） | 3 | 現在表示中のページに対応するナビ項目が色・太字等で強調表示されている（例: 管理画面内で現在のセクションが分かる） |
| 戻る導線 | 2 | 商品詳細・注文詳細等の下位ページに一覧へ戻るリンク（`← 商品一覧に戻る` 等）が表示されている |
| パンくず | 2 | 階層が2段以上あるページ（商品詳細、管理画面の各セクション等）にパンくずリストがある。無い場合、戻る導線や見出しで階層が推測できれば部分点 |
| 404/未認証時の挙動 | 3 | 存在しないURLでユーザーに分かる404画面（案内文とトップへの導線）が表示される。未ログインで会員専用ページ（カート・注文履歴等）にアクセスした際、エラーで固まらずログイン画面へ誘導される |
| 主要動線（商品発見→カート→購入）の明確さ | 3 | 商品一覧→詳細→カート追加→カート確認→注文確定までの各画面でCTAボタンが視覚的に最も目立ち、次に何をすべきか画面から即座に分かる |
| フッターの有無 | 2 | 主要ページの下部に会社情報・リンク等のフッターがある。無く余白だけで終わっている場合は減点（「作りかけ」感の直接要因） |

## 3. フィードバック・状態表示（15点）

| 項目 | 配点 | 判定基準 |
|---|---|---|
| 操作成功の伝達（カート追加・注文完了） | 4 | カートに追加した際に、ページ遷移なしで分かる成功メッセージ（例: 緑字の「カートに追加しました」）が表示される。注文確定後は専用の完了画面/メッセージが表示される |
| エラー表示のわかりやすさ | 4 | API失敗時・入力不備時にユーザーが原因を理解できる日本語メッセージが表示される。エラー文言が赤色等で視認しやすい。エラーメッセージの表示位置が原因箇所（該当フォーム要素の近く）から離れすぎていないかも確認する |
| 空状態の案内 | 3 | カートが空、注文履歴が0件、検索結果が0件などの状態で、単なる空白ではなく案内文（「カートは空です」等）と次のアクション導線がある |
| ローディング状態（コードで確認） | 2 | データ取得中にスピナー・スケルトン等の表示コードがあるか、`frontend/` の該当コンポーネントを確認する |
| 多重送信防止（コードで確認） | 2 | フォーム送信ボタンが送信処理中に `disabled` になる、または連打を防ぐ実装があるかをコードで確認する |

## 4. フォームUX（15点）

| 項目 | 配点 | 判定基準 |
|---|---|---|
| ラベル・必須表示・プレースホルダ | 4 | 全ての入力欄にラベルが表示されている。必須項目には視覚的な印（赤い`*`等）がある。分かりづらい入力（住所欄等）には入力例のプレースホルダがある。ラベル表記や必須マークの有無がフォーム間（ログイン/会員登録/管理画面モーダル等）で一貫していない場合は減点 |
| バリデーションエラーの位置と文言 | 5 | 入力不備時のエラー文言が日本語で具体的（「◯◯を入力してください」等）。エラーがフォームの直上/直下等、原因の入力欄から視覚的に近い位置に表示される。ページ最上部にだけ表示されて対象欄との対応が分かりにくい場合は減点 |
| 入力しやすさ（サイズ・タイプ） | 3 | 入力欄の高さ・幅がクリック/タップしやすい大きさ。数量選択や複数行住所欄など、入力内容に適したUIコントロール（select、textarea等）が使われている |
| 破壊的操作の確認（コードでも確認） | 3 | 削除・キャンセル等の破壊的操作について、スクリーンショット上で確認ダイアログ相当のUIが撮れていればそれで判定する。撮れていない場合は該当ボタンのイベントハンドラをコードで確認し、`window.confirm` 等の確認ステップがあるかを見る |

## 5. モバイル・レスポンシブ（20点）

| 項目 | 配点 | 判定基準 |
|---|---|---|
| モバイルでのヘッダー・ナビの使いやすさ | 4 | 375px幅でヘッダーの検索欄・ナビ項目が重なったり画面外にはみ出したりしない。ナビ項目が多い場合（ログイン後・管理者）はハンバーガーメニュー等でまとめられているか、素の折り返し表示でも視認・タップに支障がないかを確認する |
| グリッドの破綻有無 | 3 | 商品一覧等のグリッドがモバイル幅で列数調整され、カードが潰れたり重なったりしない |
| タッチターゲットサイズ | 3 | ボタン・リンク・削除アイコン等のタップ領域が指で押しやすい大きさ（目安44px角相当）になっている。文字リンクのみで極小の場合は減点 |
| 横スクロールの発生 | 3 | ページ全体が横スクロールしない（コンテンツが375px幅に収まっている）。意図的な横スクロール領域（テーブル等）以外で発生している場合は減点 |
| テーブルの扱い | 4 | 管理画面の一覧テーブル等がモバイル幅で、列内テキストが1文字ずつ縦に折り返される等の破綻を起こしていないか確認する。横スクロールコンテナ化・カード化・列の間引きなど何らかの対処があれば加点、対処がなく崩れている場合は減点 |
| 文字サイズ | 3 | モバイルで本文・価格・ボタン文字が読める大きさを保っている（デスクトップと比べて極端に縮小されていない） |

## 6. アクセシビリティ・視認性（15点）

| 項目 | 配点 | 判定基準 |
|---|---|---|
| コントラスト・文字サイズの視認性 | 4 | 文字色と背景色のコントラストが十分（淡いグレー文字が背景に埋もれていない）。本文が極端に小さいサイズになっていない |
| クリック可能要素の判別しやすさ | 4 | リンク・ボタンが色・下線・ボタン形状等で「押せるもの」だと一目で分かる。同じ見た目なのにクリック可否が異なる要素がない |
| フォーカスリング（コードで確認） | 4 | `outline: none` 等でフォーカスリングを消していないか、キーボード操作時に現在フォーカス中の要素が視覚的に分かるかをコードで確認する |
| セマンティックHTML・alt・ARIA（コードで確認） | 3 | `img` タグに意味のある `alt` があるか、モーダル・エラー表示に適切な role/aria 属性があるかをコードで確認する |

#### 第2回評価後の改善（商品画像の根本修正）

第2回評価では商品画像の文字化けが致命的な指摘として残っていたため、応急処置（`onError` フォールバック）に留まらず原因を修正した。

| 項目 | 内容 |
|---|---|
| 原因 | `backend/app/seed.py` が `placehold.co` の画像URLに日本語の商品名をそのまま渡しており、`placehold.co` が日本語グリフに対応していないため、全商品画像が豆腐文字（`?`の連続）で表示されていた |
| 修正 | 商品ごとに英数字の slug を付与し、画像URLを `https://picsum.photos/seed/<slug>/400/300`（実写画像を返すプレースホルダーサービス）に変更。`make reset` を実行してDBを作り直し、シードを再投入した |

#### 第3回評価後の改善（major指摘の修正）

第3回評価で major と判定された指摘（英語のバックエンドエラー文言、必須マークの不統一、管理画面テーブルの横スクロール発見性）を中心に修正した。

| 項目 | 内容 |
|---|---|
| バックエンドエラー文言の日本語化 | `frontend/lib/api.ts` に、重複メール登録・カート系404等の英語エラーメッセージを日本語へ変換する処理を一元化して追加 |
| 必須マークの統一 | 全フォームで「赤い`*` + `sr-only`の「（必須）」」に統一し、任意項目には「（任意）」を明示 |
| 管理画面テーブルの横スクロール発見性 | `ScrollableTable` コンポーネントを新規作成し、スクロールヒントと右端フェード表示でモバイル横スクロールの存在に気づけるようにした |
| 入力欄の高さ統一 | 入力欄の縦パディングを `py-2.5` に統一 |
| 見出し階層の是正 | `h3` になっていた見出しを `h2` に修正 |
| コントラスト改善 | `text-gray-500` を `text-gray-600` へ一括変更 |
