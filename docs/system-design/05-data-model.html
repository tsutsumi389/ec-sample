05データモデル設計

backend/app/models.py に全 17 モデル（EC 中核 10 + レコメンド・パーソナライズ 5 + アシスタント会話 2）を定義する。database.py が Base / engine / SessionLocal / get_db を提供し、seed.py が初期データを冪等投入する。全テーブルは main.py の lifespan 内 Base.metadata.create_all で自動生成される（マイグレーションツール未導入）。

5.1Product 状態機械（可視性・購入可否の唯一の源）

商品の一覧表示・商品ページ表示・購入可否はすべて Product.status の 6 状態から導出する。個別の真偽フラグは持たない（旧 is_active は archived に統合し廃止）。判定は is_listed / is_viewable / purchasable プロパティと LISTED_STATUSES / VIEWABLE_STATUSES 定数に集約される。

draft
下書き（未公開）。新規作成時のデフォルト（誤公開防止）。
listedviewablepurchasable
coming_soon
近日発売。表示はするが購入不可。
listedviewablepurchasable
on_sale
販売中。表示・購入可（在庫があれば）。
listedviewablepurchasable*
suspended
一時停止。在庫があっても表示のみで購入不可。
listedviewablepurchasable
discontinued
販売終了。一覧からは隠すが商品ページ・履歴は残す。
listedviewablepurchasable
archived
論理削除相当。一覧・商品ページとも非表示。
listedviewablepurchasable
定数含む状態用途
LISTED_STATUSEScoming_soon, on_sale, suspended一覧（トップ・検索・関連・おすすめ）に表示してよいか（is_listed）
VIEWABLE_STATUSEScoming_soon, on_sale, suspended, discontinued商品ページを直接開いて閲覧してよいか（is_viewable）。URL では見られる discontinued を含む
purchasableon_sale かつ stock > 0* 販売中かつ在庫ありのときだけ購入可能

5.2実売価格 effective_price

sale_price が設定されていればそれを、なければ price を実売価格として返す。カート小計・注文金額・OrderItem スナップショット・埋め込みテキストの価格帯言語化まで、すべて effective_price を使い price を直接参照しない。sale_price 設定時は price を定価（打ち消し表示）扱いにする。

sale_price 無し
¥2,400
effective_price = price
sale_price 有り
¥3,000 ¥2,40020% OFF
effective_price = sale_price、price は定価

5.4レコメンド・パーソナライズ用テーブル

既存テーブルにカラムを足さず新規 5 テーブルに分離している（マイグレーション未導入のため既存テーブルへの追加ができず、新規テーブル追加のみ許容される運用に合わせた設計）。うち product_views は購入・お気に入りと並ぶパーソナライズの行動シグナルで、他 4 テーブルは埋め込み・SID・生成キャッシュを担う。

テーブル主なカラム役割
product_embeddings

5.5シードデータ（seed_data）

users テーブルに 1 件でもあれば即 return する冪等ガード付き。初期投入は次のとおり。

  • ユーザー: admin@example.com（admin123, role=admin）/ user@example.com（user123, role=user）
  • カテゴリ 5 件（キッチン家電 / 生活家電 / 日用品 / アウトドア / ファッション小物）、商品 60 件。ホームのレーン分離・novelty がカテゴリ分布で多様性を測るため、5 カテゴリに配分して 1 カテゴリ偏りを避ける
  • 各状態を一通り含む（draft / coming_soon / suspended / discontinued / archived を各数件、残りは on_sale、wrist-watch=stock 0）。可視性・購入可否の分岐やレーンのフォールバックを実データで確認できる
  • 商品の description は埋め込みの原文を兼ねる。テンプレ文だと全ベクトルが密集してセマンティック検索もホームのレーン分離も効かなくなるため、用途・素材・シーン・サイズなど固有語を含む具体的な説明文にする。画像は frontend/public/products/ の統一 SVG（10 枚）を image_slug で流用（新規 SVG は追加しない）
  • クーポン 2 件: WELCOME10（percent 10）/ SAVE500（fixed 500・min 5000）
  • 動作確認用に user のサンプル注文（delivered / paid / shipped など複数ステータス、effective_price を total にスナップショット、在庫を購入分減算）とレビュー。「今週の売れ筋 Top10」レーンや注文管理のデモが成立するよう直近の購入実績を持たせ、レビュー投稿資格を満たすため実 Order/OrderItem を合わせて作る

5.6アシスタント会話テーブル

チャット型ショッピングアシスタントの会話とメッセージを永続化する新規 2 テーブル。既存テーブルにはカラムを足さず分離する（マイグレーション未導入の運用に合わせる）。会話は 1:N でメッセージを持つ（cascade="all, delete-orphan"）。会話 ID は推測困難な UUID（文字列 PK）で、未ログインでも端末の localStorage に UUID を保持することでパネル再オープン時に会話を継続できる。

テーブル主なカラム役割 / 制約
assistant_conversations
AssistantConversation		id PK（UUID 文字列）/ user_id FK users.id nullable,index（ゲスト会話は NULL）/ created_at / updated_at（onupdate）messages（1:N, cascade, id 昇順）。ログイン会話は本人のみアクセス可、ゲスト会話は UUID 保持が認可。ゲスト会話中にログインすると以降 user_id を紐付け（引き継ぎ）
assistant_messages
AssistantMessage		id PK / conversation_id FK,index / role（user / assistant）/ content / product_ids JSONB（提案商品ID配列、user 行は空）/ source nullable（llm / fallback、assistant 行のみ）/ created_atconversation（N:1）。product_ids は履歴復元時にカードを引き直すためのもので、復元時に LISTED_STATUSES を再確認し非公開化された商品は落とす

5.7商品Q&A テーブル

商品ページの購入前Q&A（§8.8）を永続化する新規 1 テーブル。既存テーブルにはカラムを足さない運用に合わせる。1 質問 = 1 AI 回答で 1 行を持ち、回答は生成時点のスナップショット（OrderItem と同じ思想で、商品説明やレビューが後から変わっても再生成しない）。

テーブル主なカラム役割 / 制約
product_questions
ProductQuestion		id PK（推測困難な UUID 文字列）/ product_id FK products.id,index / user_id FK users.id / question Text / answer Text / source（llm / fallback）/ answerable Boolean（false = 根拠不足で「情報不足」表示）/ created_at質問できるのはログインユーザーのみ（レビューと同じ権限モデル）だが、蓄積された Q&A は誰でも閲覧できる（社会的証明）。可視性は Product.is_viewable に従う
