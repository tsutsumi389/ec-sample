07主要処理フロー

7.1注文確定（カート → 在庫確認 → スナップショット → 在庫減算）

  1. POST /api/orders を認証ユーザーで呼び出し、OrderCreate（shipping_address / address_id / coupon_code）を受け取る
  2. current_user のカート明細を id 順で取得。空なら 400「カートが空です」
  3. 配送先を決定: address_id 指定時は自分の Address を整形（無ければ 404）、未指定なら shipping_address を使用。最終的に空なら 400
  4. try 開始: カート内 product_id 群を with_for_update で行ロック取得し、同時注文による超過販売（oversell）を防止
  5. 各明細を検証: 商品が存在し is_viewable か、status==on_sale か（不可なら「この商品は現在購入できません」）、stock ≥ quantity か（不足なら「在庫が不足しています」）
  6. unit_price = product.effective_price を採用。stock -= quantity で減算。total_amount に unit_price×quantity を加算
  7. OrderItem（product_name / price=unit_price / quantity）をスナップショットとして蓄積（後から商品マスタを参照しない）
  8. coupon_code があれば get_coupon_by_code → evaluate_coupon(total_amount) で検証。無効なら 400、有効なら discount_amount を確定
  9. Order を total_amount=（合計−割引）、discount_amount、coupon_code、status=pending、shipping_address、items で生成し add
  10. カート明細を全て delete し commit。例外時は rollback して再送出
  11. refresh して注文詳細（OrderDetailOut）を返す

7.2レビュー投稿

  1. get_current_user で認証（未認証は 401）
  2. 対象商品を取得。None または is_viewable でなければ 404
  3. OrderItem×Order を JOIN し、cancelled 以外の注文に当該 product_id が含まれるか確認。無ければ 403（Purchase required to review）
  4. 同一 user_id×product_id の既存 Review があれば 400（Already reviewed）
  5. Review（rating, comment）を作成し add → commit → refresh
  6. 投稿者名を含む ReviewOut を 201 で返す

7.3クーポン適用

  1. 検証: POST /api/coupons/validate に code と subtotal を送信
  2. get_coupon_by_code でコードからクーポンを取得（無ければ None）
  3. evaluate_coupon: None または is_active 偽なら「Invalid coupon code」、expires_at が現在より前なら「Coupon has expired」、subtotal < min_order_amount なら「Minimum order amount…」
  4. 全て通れば _compute_discount で算出: percent は subtotal×value//100、固定額は value。いずれも min(amount, subtotal) で上限
  5. 注文時適用: create_order 内で total_amount 確定後に同じ検証を呼び、無効なら注文中止、有効なら Order から減算し coupon_code を保存

7.4AI レコメンド生成（home API → generate_for_user）

  1. GET /api/recommendations/home。未ログインは get_popular_products（source=fallback）を返して終了
  2. ログイン時は除外 ID（購入済み＋カート内）・build_profile・RecommendationState を取得
  3. state=ready かつ profile_hash 一致なら UserRecommendation を rank 順で取得。返却時に status が LISTED か再確認し可視商品のみ source=llm で返す
  4. キャッシュ無し/陳腐化かつ profile ありかつ _should_generate（generating/failed の 10 分クールダウン外）なら生成起動
  5. _schedule_generation: pg_try_advisory_xact_lock(user_id) を試行、取れなければスキップ。取れたら mark_generating で同期的に generating を確定し BackgroundTasks に登録
  6. 即座に人気順フォールバック（除外 ID 付き）を返す
  7. レスポンス後に generate_for_user が自前セッションで起動: build_profile → get_candidates（pgvector コサイン近傍、LISTED のみ、除外、上位 20 件）
  8. 候補・履歴を SID 列挙のカタログにして system/user メッセージを構築（PII なし）
  9. ollama chat（format=JSON schema, temperature=0.3）を呼び、pydantic で検証
  10. ハルシネーション対策: 候補集合に実在する SID のものだけ採用（重複 product 除外、最大 8 件）
  11. 採用ゼロなら failed。採用ありなら旧 UserRecommendation を削除し rank 付きで差し替え、state=ready ＋ generated_at を確定。例外時は rollback し state=failed

7.5セマンティック ID 量子化（reassign_semantic_ids）

  1. pg_advisory_xact_lock（固定キー 918273645）で generation 採番と全体上書きの競合を直列化
  2. 全 ProductEmbedding を product_id 順に取得。0 件なら終了
  3. embedding を np.array（float64）化。n<2 は縮退（全商品を 0-0-0 ベースに寄せる）
  4. n≥2 は残差量子化を 3 階層ループ: k=min(8,max(2,n)) で KMeans を残差に fit_predict し labels を各階層の code に記録
  5. 残差を「元ベクトル − 割り当てセントロイド」で更新し次階層へ渡す
  6. 各商品の code3 つを a-b-c 文字列化。base 衝突時は第 3 コードに連番サフィックスで一意化
  7. 新 generation を採番し SemanticIdCodebook を add、各 ProductEmbedding に semantic_id と codebook_generation を刻む
  8. 最新 generation 以外のコードブックを削除（肥大化防止）し commit

7.6認証（トークン検証）

  1. login で create_access_token が payload {sub: user_id, iat, exp: +24h} を HS256 署名
  2. 各リクエストで HTTPBearer が Authorization ヘッダを受け取り decode_access_token で検証（不正・期限切れは 401）
  3. _resolve_user_from_token: sub から user_id を取り出し db.get(User)。不在なら 401
  4. password_changed_at が設定済みなら iat と比較。iat が無いか変更時刻より前なら 401（トークン失効）
  5. 必須認証は get_current_user、任意認証（get_current_user_optional）は無効トークンでも None 匿名扱い、管理者は get_current_admin で role==admin を確認

7.7アシスタントのチャット処理（同期パス・500 を返さない）

  1. POST /api/assistant/chat を任意認証で受け、AssistantChatIn（conversation_id? / message 1〜500 字）を検証
  2. 会話の取得/作成: conversation_id=null なら UUID で新規 AssistantConversation を作成（ログイン中なら user_id 付与）、flush で id 確定。既存なら _load_conversation で所有チェック（他人/不在は 404）し、ゲスト会話中にログイン済みなら user_id を紐付け（引き継ぎ）
  3. 既存会話はメッセージ数を数え、50 件以上なら 400（会話の肥大化防止）。過去メッセージを (role, content) の古い順でロード
  4. ユーザーメッセージを AssistantMessage（role=user、product_ids=[]）として add（未 commit）
  5. generate_reply を呼ぶ。ここは例外を握って必ず結果を返す（500 を出さない）
  6. 候補抽出（ハイブリッド）: build_query_text で直近 3 ユーザー発話を改行連結して埋め込み（embeddinggemma）→ pgvector コサイン近傍 top-20（LISTED のみ）。加えて新メッセージをトークン分割した ILIKE キーワード検索 top-10。重複除去してベクトル優先でマージ。候補ゼロならフォールバックへ
  7. プロンプト構築: 候補を llm_catalog.catalog_line（SID / 商品名 / カテゴリ / effective_price / 平均★）で列挙し、履歴（直近 6 ターン・各 200 字）と <message> で囲んだ新メッセージを付す。ログインユーザーは行動履歴（weight 上位 10 件、§8.5）を【お客様のこれまでの行動】として前置きする。PII は入れない
  8. Ollama chat（format= JSON schema, temperature=0.3, timeout 60 秒）で構造化出力 {reply, items:[{sid, reason}]} を取得・pydantic 検証
  9. SID 検証: llm_catalog.match_products で候補マップに実在する SID のものだけ採用（正規化・重複除去・最大 4 件）。strip_sids_from_reply で reply 本文の SID 表記を正規表現で除去。本文が空なら定型文に落とす。source=llm で返す
  10. フォールバック分岐: Ollama 接続不可・タイムアウト・parse 失敗・候補ゼロのいずれでも、キーワード検索 top-4 + 定型文を source=fallback で返す。キーワード 0 件なら get_popular_products top-4（人気順）で必ず商品を出す。Ollama 例外は DB を汚さないため rollback しない
  11. assistant メッセージ（content=reply、product_ids=採用商品ID、source）を add して commit。採用商品を RecommendationItemOut に整形して AssistantChatOut を返す
