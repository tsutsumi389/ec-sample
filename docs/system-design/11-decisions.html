11重要な設計上の決定・運用ルール

コードに現れにくい暗黙の前提を含め、本システムを貫く設計判断をまとめる。改修時はこれらを厳守すること。

決定内容 / 理由
マイグレーション未導入Alembic 等は入れない。テーブルは main.py lifespan の Base.metadata.create_all で自動生成（OperationalError で最大 10 回リトライ）。モデル変更は make reset で DB 作り直さないと反映されない
既存テーブルへの追加は不可マイグレーション未導入のため、レコメンド機能は既存テーブルにカラムを足さず新規 5 テーブル（埋め込み・SID・キャッシュ 4 ＋ 閲覧履歴 product_views）に分離。将来の拡張も新規テーブル追加で行う
パーソナライズ信号の分離と時間減衰閲覧履歴は専用テーブル product_views（1 ユーザー×1 商品 1 行の upsert）に持ち既存テーブルを汚さない。行動重みは半減期 30 日で時間減衰させ「今の好み」を優先。profile_hash には 種別:product_id の集合のみを入れ重み・時刻は含めない（時間経過だけでキャッシュが陳腐化しないため）。§8.6
商品は論理削除のみProduct を物理削除してはならない。status="archived" にする。過去注文のスナップショットは不変に保たれる
status が可視性・購入可否の唯一の源6 状態から is_listed/is_viewable/purchasable を導出。個別の真偽フラグを増やさない（旧 is_active は archived に統合）
実売価格は effective_pricesale_price があればそれ、なければ price。カート小計・注文金額・OrderItem スナップショットは全て effective_price を使い price を直接使わない
注文明細はスナップショットOrderItem は注文時点の product_name / price を保持。Order も coupon_code を文字列で、discount_amount / total_amount を確定値で持つ。商品マスタを再参照しない
在庫整合性注文確定・キャンセルとも対象商品行を with_for_update で行ロックしてから在庫を増減し、同時実行による超過販売を防止
リソース所有権の限定カート・注文・住所の各操作は必ず user_id == current_user.id でフィルタし、他人のリソースは 404 で隠蔽
API プレフィックス全ルートは /api 配下（main.py で一括登録）。admin ルーターは実効パス /api/admin/...
CORS許可オリジンは http://localhost:3000 のみ（allow_credentials=True、methods/headers は全許可）
耐障害起動pgvector 拡張が使えない環境では product_embeddings を作成除外。Ollama が使えなくても埋め込み同期をスキップし警告ログのみで起動継続
非同期処理商品作成/更新後の埋め込み更新、LLM レコメンド生成は BackgroundTasks で非同期実行し、失敗を本体処理に影響させない
検索サジェストは軽量クエリのみ/api/products/suggest は入力中の高頻度呼び出しに耐えるため、LISTED 商品名への ILIKE（エスケープ済み）だけで応答し、埋め込み・レビュー集計等の重い処理は一切呼ばない。2 文字未満は DB も引かず即空返し。セマンティック検索は確定後の GET /products?search= だけが担う
アシスタント API は 500 を返さない/api/assistant/chat は Ollama 接続不可・タイムアウト・parse 失敗・候補ゼロのいずれでも例外を握り、キーワード検索（0 件なら人気順）＋定型文の source="fallback" で必ず 200 応答する。レコメンドの「絶対に止めない」思想の踏襲。商品Q&A（POST /products/{id}/questions）も同じ思想でフォールバック定型文を保存して必ず 201
Q&A 回答はスナップショットproduct_questions の回答は生成時点のまま永続化し、商品説明・レビューが後から変わっても再生成しない（OrderItem と同じ思想）。既存テーブルにカラムは足さず新規 1 テーブルのみ追加。answerable=false（根拠不足）も正直にそのまま保存・公開する
アシスタントは同期パスレコメンドと異なりユーザーが応答を待つ対話のため、候補抽出・LLM 生成をリクエスト同期パスで実行する（threadpool）。将来は ollama.AsyncClient 化で解消可能
ゲスト会話の認可ログイン会話は本人のみ。ゲスト会話（user_id NULL）は UUID を知っていること自体が認可（サンプルとして許容）。本番想定なら署名付きセッション（HttpOnly Cookie 等）でゲスト会話を端末に束縛する必要がある旨をコードコメントに明記
reply 本文に SID を出さないsystem プロンプトで「本文に SID を書かない」と指示した上で、防御的に strip_sids_from_reply の正規表現後処理で SID x-y-z 表記を除去（小型モデルはプロンプト指示だけでは漏れが残るため二段構え）
SID 照合・カタログ整形の共通化ハルシネーション対策の SID→Product 照合とカタログ 1 行表現は services/llm_catalog.py に集約し、レコメンドとアシスタントで共用（重複実装しない）
ホームは 1 レーン = 1 アルゴリズム各ビルダーは HomeContext から候補レーンを返すだけの純粋関数で、ページ全体の都合を知らない。1 本のレーンが複数アルゴリズムの混合物になることを型と BUILDERS レジストリで構造的に禁じる。レーン追加はビルダー関数の追加のみで行う（§8.7）
ホームは stage-wise 貪欲法で組むrow-ranking ではなく「既出レーン・既出商品」込みでスコアし 1 本選ぶたびに再スコアする。score = relevance × prior × novelty（乗算で 1 要素が致命的に低い棚の救済を防ぐ）。重複排除は目的関数に埋めず明示フィルタ工程にして「同一商品が 2 セクションに出ない」を保証にする。max_lanes 上限より先に score 閾値 0.20 で打ち切ることを許容
ホームは何があっても 200pgvector 不在・Ollama 停止・埋め込み 0 件・プロフィール構築不能でも、載せられるレーンだけでページを組んで返す。_safe / _safe_build で各処理を隔離し失敗時は必ず rollback（残すと以後同一セッションが InFailedSqlTransaction で連鎖失敗）。コールドスタートは新着レーンが最後の砦
プロフィールは 1 リクエスト 1 回ホームの全ビルダーが共有するプロフィール・除外 ID・LLM キャッシュ鮮度は build_context で 1 回だけ構築して使い回す（ビルダーごとに build_profile を呼ぶとレーン本数ぶん重複実行される）。ゲストは recently_viewed_ids の位置減衰加重平均で擬似プロフィールを作りパーソナライズする
冪等性seed_data は users が非空なら即 return。お気に入り追加は重複時に既存返却、削除は非存在時 204
テスト未整備テストコードは無い。追加する場合は backend/（pytest 想定）から
