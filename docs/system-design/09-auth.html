09認証・認可設計

JWT（HS256、Bearer トークン）によるステートレス認証を提供し、bcrypt でパスワードをハッシュ化する。トークンは user_id を sub に持ち、有効期限は 24 時間。サーバ側トークンストアやブラックリストは持たない。

9.1トークンと依存性

依存性挙動
get_current_user必須認証。HTTPBearer(auto_error=True) でヘッダを要求し、検証済み User を返す
get_current_user_optional任意認証。HTTPBearer(auto_error=False)。ヘッダ無し・無効・期限切れ・失効・削除済みでも 401 を投げず None（匿名）を返す。レコメンド等のフォールバック用
get_current_admin管理者認可。get_current_user に依存し role≠admin なら 403（Admin privileges required）

9.2パスワード変更時の全セッション失効

  1. update_password で current_password を検証（不一致は 400）
  2. new_password を bcrypt で保存し、password_changed_at を秒精度（microsecond=0）に丸めて記録
  3. 以後 _resolve_user_from_token でトークンの iat が password_changed_at より前なら 401 にして失効させる
  4. 秒に丸めるのは、変更直後の再ログインで得た同一秒 iat の新トークンが誤失効しないため

9.3セキュリティ上の取り決め

  • login はメール不在・パスワード不一致を区別せず一律 401「Incorrect email or password」（ユーザー列挙防止）
  • register は既存メールを 400 で拒否。新規ユーザーは常に role=user（管理者昇格の API 経路は無い）
  • 認証失敗時の 401 応答には常に WWW-Authenticate: Bearer ヘッダを付与
  • 認可は role 文字列で判定（user / admin）。細粒度の権限は無い
  • リフレッシュトークンの仕組みは無い（有効期限 24h 固定）
設計リスク

JWT 署名鍵 SECRET_KEY は環境変数から取得するが、未設定時は "dev-secret-key-change-in-production" というハードコードのデフォルト値にフォールバックする。本番で環境変数未設定だと既知の弱鍵で署名される。
